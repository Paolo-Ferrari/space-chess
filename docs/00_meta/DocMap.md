# Doc Map — карта документации CyberChess: Overclock Protocol

Полное описание структуры `docs/` для студийной разработки (дизайн + инженерия).  
Код в этом задании не пишется; содержание игровых чисел — только после утверждения авторов.

---

## Как читать карту

Для каждого документа ниже:

1. **Цель**  
2. **Разделы внутри**  
3. **Что записываем**  
4. **Когда используем**  
5. **Зависимости** (от чего зависит / кто зависит от него)  
6. **Постоянное обновление?**  

Статусы файлов сейчас: `CHARTER` = каркас утверждён, тело наполняется итеративно.

# Обязательные документы

## Project Vision  
`00_overview/ProjectVision.md`

1. **Цель:** Зачем существует продукт, для кого, какой опыт, границы «не делаем».  
2. **Разделы:** Elevator pitch; целевая аудитория; столган; USP; non-goals; критерии успеха (продуктовые, не KPI-выдумки без данных).  
3. **Что записывать:** Утверждённое видение руководства/GD; без таблиц юнитов.  
4. **Когда:** Старт проекта, питчи, онбординг, проверка scope фич.  
5. **Зависимости:** База для GDD, Roadmap, UI Flow. Зависит от бизнес-решений владельца.  
6. **Обновление:** Редко (мажорные смены направления).

## Game Design Document (GDD)  
`01_design/GameDesignDocument.md`

1. **Цель:** Главный индекс дизайна: ссылки на профильные docs + краткая целостная картина.  
2. **Разделы:** Обзор жанра; ссылки на Rules / Combat / Factions / …; MVP vs Target; открытые вопросы.  
3. **Что записывать:** Сводку и ссылки; детали — в дочерних docs (не дублировать 50 страниц).  
4. **Когда:** Дизайн-синк, приёмка фич («есть ли в GDD?»).  
5. **Зависимости:** Vision ←; → все `01_design` и `02_content`.  
6. **Обновление:** Да, при каждом утверждении крупного дизайн-блока.

## Domain Model  
`04_engineering/DomainModel.md`

1. **Цель:** Каталог сущностей и связей (Definition vs Instance).  
2. **Разделы:** Слои Catalog/Meta/Match; каждая сущность; инварианты.

## Packages Architecture

2. **Разделы:** Деревья пакетов; запреты; граф зависимостей.  
3. **Что записывать:** Куда класть новый модуль ядра/контента.

5. **Зависимости:** ← Architecture; → FE/BE.

## Game Loop  
`01_design/GameLoop.md`

1. **Цель:** Циклы игрока: сессия, матч, ход, мета (армии).  
2. **Разделы:** Core loop; session loop; match loop; turn loop; выход/реванш.  
3. **Что записывать:** Последовательности состояний без изобретения новых механик.  
4. **Когда:** Проектирование экранов и state machine.  
5. **Зависимости:** ← GDD, Rules; → Game States, UI Flow, Frontend Architecture.  
6. **Обновление:** Да, при смене петли.

## Game Rules  
`01_design/GameRules.md`

1. **Цель:** Нормативные правила партии (ход, победа, сдача, база движения).  
2. **Разделы:** Старт; очередь ходов; действие за ход; победа/поражение; запреты; краевые случаи.  
3. **Что записывать:** Только утверждённые правила (сейчас: ортогональный ход 1, одно действие → передача хода, короли/командир, сдача).  
4. **Когда:** Реализация движка, тестирование, споры «а можно ли».  
5. **Зависимости:** ← GDD; → Combat, Game States, Testing.  
6. **Обновление:** Да, при любом изменении правил.

## Combat System  
`01_design/CombatSystem.md`

1. **Цель:** Как разрешаются атака, урон, способности в бою.  
2. **Разделы:** Базовая атака (TBD форма); урон/HP; способности Factor 2; смерть фигуры; взаимодействие со щитами/дебаффами (когда появятся).  
3. **Что записывать:** Утверждённые формулы и порядок резолва; **не выдумывать** дистанцию атаки до решения.  
4. **Когда:** Реализация combatRules / abilities.  
5. **Зависимости:** ← Rules, Units, Implants; → Balance, Testing.  
6. **Обновление:** Да.
