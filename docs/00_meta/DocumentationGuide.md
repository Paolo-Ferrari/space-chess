# Documentation Guide

| Поле | Значение |
|------|----------|
| Status | Active |
| Owner | Tech Lead + Lead Game Designer |
| Audience | Вся команда (до ~20 человек, годы разработки) |

## Цель

Единые правила работы с `docs/`: где искать правду, как обновлять, что нельзя придумывать «в коде раньше docs».

## Разделы

1. Иерархия источников правды  
2. Статусы документов (`CHARTER` / `DRAFT` / `APPROVED` / `DEPRECATED`)  
3. Процесс изменения (PR + ревью владельца документа)  
4. Именование и язык  
5. Связь с Git / релизами  

## Что записывать

- Кто утверждает дизайн vs архитектуру  
- Запрет: механики/баланс/юниты вне утверждённых docs  
- Как онбордить нового разработчика (читать README → Vision → GDD → Architecture)

## Когда используется

Постоянно: онбординг, ревью PR, споры о scope.

## Зависимости

Все документы `docs/` подчиняются этому гайду.

## Обновление

Да, при смене процесса команды.
